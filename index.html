      Prompt Engineering

      Guide du Prompt Engineering

      Un support de lecture complet pour apprendre à dialoguer efficacement avec les intelligences artificielles génératives — sans prérequis.

      Introduction

      Objectifs pédagogiques

      À la fin de ce guide, vous serez capable de :

      • comprendre ce qu'est un prompt ;
      • comprendre le rôle du Prompt Engineering ;
      • rédiger un prompt structuré ;
      • dialoguer efficacement avec une IA ;
      • améliorer progressivement vos prompts ;
      • connaître les principaux outils d'IA générative.

      Qu'est-ce que l'IA générative ?

      L'intelligence artificielle générative désigne des modèles capables de produire du contenu nouveau : texte, code, images, audio. Contrairement aux systèmes qui se limitent à classer ou prédire, ces modèles génèrent une réponse à partir d'une instruction formulée en langage naturel — le prompt.

      Les modèles de langage (LLM) comme GPT, Claude ou Gemini ont été entraînés sur d'immenses corpus. Ils prédisent le prochain token le plus probable compte tenu du contexte. Résultat : ils peuvent expliquer un concept, écrire une fonction, reformuler un e-mail ou proposer une architecture — à condition d'être bien guidés.

      Pourquoi apprendre le Prompt Engineering ?

      Le Prompt Engineering est l'art et la méthode de formuler des instructions claires, structurées et contextuelles pour obtenir le meilleur résultat d'une IA. Ce n'est pas une « magie » : c'est une compétence de communication technique, proche de la rédaction d'un cahier des charges ou d'une user story.

      Sans méthode, on obtient des réponses vagues, incomplètes ou incorrectes. Avec une méthode, on gagne du temps, on réduit les allers-retours et on produit des livrables exploitables en formation et en entreprise.

      Bénéfices dans le monde professionnel

      Productivité

      Accélérer la rédaction de code, de documentation, de tests et de spécifications.

      Qualité

      Obtenir des sorties structurées, vérifiables et adaptées au contexte métier.

      Montée en compétences

      Utiliser l'IA comme tuteur pour comprendre un stack, un bug ou un concept.

      Métiers concernés

      Le Prompt Engineering touche presque tous les métiers du numérique : développement (front, back, full stack), UX/UI, product ownership, scrum, formation, cybersécurité (pentester, SOC), DevOps, administration système, data, marketing, rédaction web et gestion de projet.

      Principaux modèles d'IA

      ChatGPT (OpenAI) est un assistant conversationnel polyvalent, très utilisé pour le code, la rédaction et l'explication de concepts. Les modèles GPT se distinguent par leur capacité de raisonnement et leur écosystème (API, plugins, agents).

      Claude (Anthropic) excelle souvent sur les longs contextes, l'analyse de documents et la rédaction technique soignée. Fréquemment intégré dans des environnements de développement assisté.

      Gemini (Google) s'intègre à l'écosystème Google et propose des capacités multimodales (texte, image). Utile pour la productivité bureautique et certaines tâches de code.

      Cursor, GitHub Copilot et d'autres assistants IDE intègrent des LLM directement dans le flux de développement : complétion, chat contextuel, édition multi-fichiers. Le prompt y est souvent implicite (fichier ouvert, sélection, conventions du dépôt).

      Conseil de lecture

      Parcourez les chapitres dans l'ordre la première fois. Ensuite, utilisez le menu, la recherche et la bibliothèque de prompts (chapitre 11) comme référence quotidienne.

      Chapitre 1 — Qu'est-ce qu'un prompt ?

      Un prompt est l'instruction que vous donnez à une IA pour obtenir une réponse utile.

      Définition

      Un prompt est un message (souvent en langage naturel) qui décrit ce que vous attendez du modèle : une explication, du code, un plan, une correction, etc. Il peut être une phrase courte ou un brief détaillé de plusieurs paragraphes.

      Fonctionnement (vue simplifiée)

      Votre intention → Prompt rédigé → Modèle (LLM) → Réponse générée → Vous vérifiez

      Le modèle ne « comprend » pas comme un humain : il produit la suite la plus probable. Votre prompt conditionne fortement cette suite. D'où l'importance du contexte, des contraintes et du format.

      Les ingrédients d'un prompt

      Contexte
      Qui vous êtes, où vous intervenez, quelles sont les données de départ (stack, public, contraintes projet).
      Instructions
      Ce que le modèle doit faire concrètement (expliquer, générer, comparer, refactoriser…).
      Contraintes
      Ce qu'il ne doit pas faire, les limites techniques, le niveau de langage, la longueur.
      Résultat attendu
      La forme de la sortie : liste, code, tableau, JSON, étapes numérotées…

      Exemples progressifs

      Niveau 1 — trop vague

      Exemple faible 
      Explique JavaScript.

      Niveau 2 — mieux ciblé

      Exemple intermédiaire 
      Explique JavaScript à un débutant CDA en 8 puces, avec un exemple de variable et de fonction.

      Niveau 3 — structuré

      Exemple recommandé 
      Rôle : formateur CDA bienveillant.
Contexte : apprenant débutant, aucun prérequis JS.
Objectif : expliquer ce qu'est JavaScript et à quoi il sert dans une page web.
Contraintes :
- français simple ;
- maximum 200 mots ;
- 1 analogie du quotidien ;
- 1 mini exemple de code commenté.
Format : titre court, puis paragraphes, puis exemple.
      À retenir

      Plus votre prompt précise le qui, le quoi, le comment et le sous quelle forme, plus la réponse sera actionnable.

      Chapitre 2 — Anatomie d'un bon prompt

      Cinq blocs suffisent pour structurer la quasi-totalité des demandes professionnelles.

      1
      Rôle Qui doit « incarner » le modèle ? (ex. développeur Front-End senior, PO, formateur)
      2
      Contexte Projet, public, stack, fichiers déjà existants, contraintes métier.
      3
      Objectif La mission unique et claire à accomplir dans cette réponse.
      4
      Contraintes Technologies autorisées, style, longueur, interdits (pas de CDN, pas de framework…).
      5
      Format de sortie Code prêt à coller, liste, tableau Markdown, JSON, checklist…

      Exemple complet annoté

      Template R-C-O-C-F 
      Rôle : Développeur Front-End Senior (HTML5, CSS3, JS ES6), pédagogue.
Contexte : guide de formation CDA ouvert via index.html, sans framework ni CDN.
Objectif : proposer la structure HTML sémantique d'une section « Anatomie d'un prompt ».
Contraintes :
- accessibilité (titres hiérarchisés, textes alternatifs si images) ;
- mobile first ;
- aucun framework.
Format : bloc HTML commenté + courte explication (5 lignes max).

      Demander « explique Docker » à un formateur débutant ou à un SRE ne produit pas le même niveau de détail, ni le même vocabulaire. Le rôle oriente le ton, la profondeur et les priorités.

      Si vous devez coller la réponse dans un README, demandez du Markdown. Si vous devez l'injecter dans un script, demandez du JSON valide. Le format fait partie du contrat.

      Chapitre 3 — Les bonnes pratiques

      Six réflexes pour passer d'une demande floue à un prompt professionnel.

      1. Être précis

      Remplacez les adjectifs vagues (« bien », « moderne », « rapide ») par des critères mesurables : stack, longueur, public, critères d'acceptation.

      ❌ Mauvais prompt
      À éviter 
      Fais un site moderne.

      Trop large : le modèle invente le brief.

      ✅ Bon prompt
      Recommandé 
      Crée une page d'accueil HTML/CSS pour un portfolio développeur CDA.
Style : épuré, mobile first, palette fond #F4F3EE / texte #463F3A.
Sections : hero, projets (3), contact.
Pas de framework ni CDN.

      Critères concrets : le modèle peut livrer quelque chose d'évaluable.

      2. Fournir du contexte

      Indiquez le niveau de l'audience, les fichiers déjà présents, les contraintes scolaires ou d'entreprise.

      Contexte projet 
      Contexte : dépôt déjà structuré (css/, js/, index.html).
Les variables de couleur existent dans variables.css.
N'introduis pas de nouvelles dépendances.
Objectif : ajouter un composant accordéon accessible dans components.css + navigation.js.

      3. Définir clairement un objectif

      Une mission principale par prompt. Si vous avez trois objectifs, faites trois prompts (ou une liste numérotée explicitement priorisée).

      4. Imposer un format de réponse

      Contrat de format 
      Format de sortie obligatoire :
1. Résumé (3 lignes)
2. Code complet
3. Checklist de vérifications (5 items)
Aucun autre texte hors de cette structure.

      5. Découper une demande complexe

      Pour une application complète : d'abord l'architecture, puis le HTML, puis le CSS, puis le JS. Chaque étape s'appuie sur la précédente.

      1. Demander le plan des fichiers et responsabilités.
      2. Générer le HTML sémantique.
      3. Styler mobile first.
      4. Ajouter l'interactivité (navigation, copie, recherche).
      5. Passer une revue accessibilité.

      6. Améliorer par itérations

      Traitez le premier jet comme un brouillon. Ensuite : « garde la structure, simplifie le vocabulaire », « ajoute 2 exemples », « corrige l'accessibilité clavier ».

      Itération 
      Améliore ta réponse précédente :
- conserve la structure HTML ;
- réduis le CSS de 20 % sans perdre le responsive ;
- ajoute les attributs ARIA manquants ;
- explique chaque changement en une phrase.
      Méthode SCRIBE (aide-mémoire)

      Situation · Cible · Rôle · Instructions · Bornes (contraintes) · Exigence de format.

      Chapitre 4 — Les erreurs fréquentes

      Comparer un mauvais et un bon prompt entraîne l'œil plus vite qu'un long discours.

      Manque de contexte

      ❌ Mauvais prompt
      À éviter 
      Corrige mon code.

      Aucun extrait, aucun symptôme.

      ✅ Bon prompt
      Recommandé 
      Voici une fonction JS qui doit filtrer une liste de prompts par catégorie.
Bug : les accents cassent le filtre.
Stack : vanilla ES6, navigateur moderne.
Colle le correctif + 1 test manuel à faire.

      Contexte + symptôme + livrable attendu.

      Demande trop vague

      ❌ Mauvais prompt
      À éviter 
      Parle-moi de la cybersécurité.
      ✅ Bon prompt
      Recommandé 
      Pour un étudiant CDA débutant, explique la notion de surface d'attaque web en 6 puces,
avec 1 exemple concret (formulaire de login) et 1 contre-mesure.

      Plusieurs objectifs mélangés

      ❌ Mauvais prompt
      À éviter 
      Écris mon API, mon README, mes tests et mon Dockerfile.

      Trop de chantiers = résultats superficiels.

      ✅ Bon prompt
      Recommandé 
      Étape 1/4 — Architecture uniquement.
Propose l'arborescence et les responsabilités des modules pour une API Express de gestion de prompts.
Pas de code détaillé encore. Format : liste de fichiers + 1 phrase chacun.

      Un objectif clair, les suivants viendront ensuite.

      Absence de contraintes

      ❌ Mauvais prompt
      À éviter 
      Fais une interface de guide.
      ✅ Bon prompt
      Recommandé 
      Crée l'UI d'un guide pédagogique.
Contraintes non négociables : HTML/CSS/JS only, pas de framework, pas de CDN, pas de backend,
palette fond #F4F3EE, texte #463F3A, accent #BCB8B1, mobile first, accessible clavier.

      Absence de format attendu

      ❌ Mauvais prompt
      À éviter 
      Donne-moi des bonnes pratiques Git.
      ✅ Bon prompt
      Recommandé 
      Liste 8 bonnes pratiques Git pour étudiants CDA.
Format : tableau Markdown | Pratique | Pourquoi | Exemple de commande |.
      Piège classique

      Accepter la première réponse sans la relire. L'IA peut halluciner des API, des chemins de fichiers ou des commandes. La vérification humaine reste obligatoire — surtout en cybersécurité et en production.

      Chapitre 5 — Les rôles

      Assigner un rôle professionnel oriente le vocabulaire, les priorités et le niveau de détail de la réponse.

      Voici des amorces prêtes à l'emploi pour les métiers du numérique. Adaptez le contexte (projet, stack, public) avant d'envoyer.

      Développeur Front-End 
      Tu es développeur Front-End senior (HTML/CSS/JS). Propose un composant d'onglets accessible WAI-ARIA, sans framework, avec explication pédagogique pour un CDA.

      Développeur Back-End 
      Tu es développeur Back-End Node/Express. Concevois une route POST /api/prompts avec validation des champs title et category, réponses HTTP correctes, sans base de données (mémoire).

      Développeur Full Stack 
      Tu es full stack. Explique comment relier un front statique à une API REST locale : endpoints, CORS en dev, gestion d'erreurs côté fetch. Schéma + étapes.

      UX/UI Designer 
      Tu es UX/UI designer. Produis les microcopy et la hiérarchie visuelle d'un écran « bibliothèque de prompts » (filtres, états vide/chargement/erreur). Pas de mockup Figma : spécification textuelle claire.

      Product Owner 
      Tu es Product Owner. Rédige 3 user stories INVEST pour un guide de Prompt Engineering (recherche, progression, mode sombre) avec critères d'acceptation Given/When/Then.

      Scrum Master 
      Tu es Scrum Master. Anime une rétrospective sur l'adoption de l'IA dans une équipe étudiante : 5 questions, format 45 min, règles de bienveillance.

      Formateur 
      Tu es formateur CDA. Construis un déroulé de 2 h sur « Anatomie d'un prompt » : objectifs, activités (sans quiz technique), supports, évaluation formative par observation.

      Pentester 
      Tu es pentester web. Explique comment formuler des prompts pour générer des checklists OWASP sans demander d'exploits offensifs prêts à l'emploi. Exemples défensifs uniquement.

      Analyste SOC 
      Tu es analyste SOC. Propose un prompt pour résumer un ticket d'alerte (chronologie, IOC, impact, actions) au format fiche incident d'une page.

      DevOps 
      Tu es DevOps. Rédige un prompt pour obtenir un pipeline CI (lint+test) GitHub Actions pour un site statique HTML/CSS/JS, avec cache et Node pour les tests éventuels.

      Administrateur Système 
      Tu es administrateur système Linux. Demande à l'IA un runbook de sauvegarde d'un dossier /var/www/guide : rotation, permissions, vérification d'intégrité. Niveau débutant encadré.

      Data Analyst 
      Tu es data analyst. Crée un prompt pour faire nettoyer un CSV de tickets (colonnes date, type, priorité) et produire des stats descriptives + insights en français.

      Marketing 
      Tu es marketing digital B2B. Rédige un prompt pour un email de lancement d'une formation Prompt Engineering (objet A/B, corps 120 mots, CTA).

      Rédacteur Web 
      Tu es rédacteur web SEO. Produis un plan Hn pour un article « Prompt Engineering pour développeurs » (intention informationnelle, 1200 mots cibles).

      Chef de Projet 
      Tu es chef de projet digital. Établis un planning 3 semaines pour livrer un guide HTML pédagogique : jalons, risques, RACI simplifié.
      Astuce

      Combinez rôle + contraintes d'éthique : « n'invente pas de CVE », « cite tes incertitudes », « propose une alternative plus simple pour un junior ».

      Chapitre 10 — Concepts avancés

      Ces notions prolongent le Prompt Engineering vers des usages plus robustes et automatisés. Chaque fiche reste volontairement accessible.

      Zero-shot Prompting

      Définition. Demander une tâche à l'IA sans fournir d'exemple, uniquement la consigne.

      Consigne→Modèle→Réponse

      Exemple concret

      Zero-shot Prompting 
      Classifie le sentiment de ce message (positif, neutre, négatif).
Message : "Le déploiement s'est bien passé, bravo à l'équipe."
Réponds par un seul mot.
      Avantages
      • Rapide à écrire
      • Peu de tokens consommés
      • Idéal pour tâches courantes
      Limites
      • Moins fiable sur formats rares
      • Style moins contrôlé
      Cas d'usage
      • Reformulation
      • Traduction
      • Résumé simple
      • Classification basique

      Few-shot Prompting

      Définition. Fournir 2 à 5 exemples avant la vraie requête pour guider le format et le style.

      Exemples+Consigne→Modèle→Réponse calquée

      Exemple concret

      Few-shot Prompting 
      Exemples :
Entrée : "Le bouton ne répond pas" → bug
Entrée : "Ajouter un export CSV" → feature
Entrée : "C'est quoi un JWT ?" → question
Classe : "Impossible de me connecter depuis Safari" →
      Avantages
      • Meilleure conformité au format
      • Aligne le ton
      • Réduit les ambiguïtés
      Limites
      • Consomme du contexte
      • Exemples biaisés ⇒ sorties biaisées
      Cas d'usage
      • Classification de tickets
      • Extraction d'entités
      • Style éditorial

      Chain of Thought

      Définition. Demander à l'IA de raisonner étape par étape avant la réponse finale.

      Problème→Étape 1→Étape 2→Conclusion

      Exemple concret

      Chain of Thought 
      Analyse ce bug en suivant strictement :
1) symptômes 2) causes possibles 3) vérifications 4) correctif 5) risques.
Code : {snippet}
      Avantages
      • Meilleure qualité sur problèmes complexes
      • Debugging plus transparent
      Limites
      • Réponses plus longues
      • Pas une garantie de vérité
      Cas d'usage
      • Debug
      • Architecture
      • Diagnostic
      • Problèmes multi-contraintes

      Structured Output

      Définition. Imposer un schéma de sortie (JSON, Markdown, tableau) pour faciliter l'automatisation.

      Prompt + schéma→Modèle→JSON / tableau

      Exemple concret

      Structured Output 
      Réponds UNIQUEMENT en JSON valide (aucun texte autour) :
{"summary": string, "risks": string[], "nextSteps": string[]}
Sujet : mise en production d'un guide statique vendredi.
      Avantages
      • Intégrable dans un pipeline
      • Facile à valider
      • Moins de rework
      Limites
      • Rigidité
      • Validation JSON toujours nécessaire
      Cas d'usage
      • APIs
      • Agents
      • Génération de configs
      • Tableaux de suivi

      RAG (Retrieval Augmented Generation)

      Définition. Enrichir le prompt avec des documents récupérés (syllabus, docs internes) avant génération.

      Question→Recherche docs→Prompt enrichi→Réponse citée

      Exemple concret

      RAG (Retrieval Augmented Generation) 
      Tu réponds UNIQUEMENT avec le contexte fourni.
Contexte : """{documents}"""
Question : {q}
Si l'information est absente, réponds : "Information non trouvée dans le corpus."
Cite le passage utilisé.
      Avantages
      • Réponses ancrées dans vos sources
      • Réduit certaines hallucinations
      • Idéal pour FAQ métier
      Limites
      • Qualité dépend de la recherche
      • Nécessite un corpus propre
      Cas d'usage
      • Documentation interne
      • Syllabus CDA
      • Base de connaissance support

      Function Calling

      Définition. L'IA choisit d'appeler des fonctions déclarées (outils) avec des arguments structurés.

      User→LLM→Appel outil→Résultat→Réponse

      Exemple concret

      Function Calling 
      Outils disponibles :
- getWeather(city: string)
- createTicket(title: string, priority: "low"|"medium"|"high")
Question : "Il pleut à Lyon et mon VPN est down, ouvre un ticket urgent."
Réponds par l'appel d'outil JSON approprié.
      Avantages
      • Actions réelles (API, tickets)
      • Sorties machine-readable
      Limites
      • Conception des outils critique
      • Besoin de validation serveur
      Cas d'usage
      • Assistants métier
      • Automatisation
      • Orchestration API

      MCP (Model Context Protocol)

      Définition. Protocole standardisant la connexion d'un modèle à des outils et sources de contexte externes.

      Agent / IDE↔MCP↔Outils & données

      Exemple concret

      MCP (Model Context Protocol) 
      Explique à un étudiant CDA, en analogie USB pour logiciels,
comment un agent IDE utilise MCP pour exposer un outil search_docs
sans hardcoder chaque intégration. 1 schéma ASCII + 5 phrases.
      Avantages
      • Interopérabilité
      • Réutilisation d'intégrations
      • Séparation agent / outils
      Limites
      • Écosystème encore jeune
      • Gouvernance des accès à prévoir
      Cas d'usage
      • Agents IDE
      • Connexions Notion/Figma/DB
      • Outils internes entreprise

      Agents IA

      Définition. Systèmes qui planifient, utilisent des outils et itèrent vers un objectif avec peu de supervision.

      Objectif→Plan→Outils→Itération→Livrable

      Exemple concret

      Agents IA 
      Tu es un agent de refactoring prudent.
Objectif : extraire le module clipboard du monolithe app.js.
Règles : diffs minimaux, ne pas casser l'API publique, résumer les fichiers touchés.
Planifie puis exécute étape par étape.
      Avantages
      • Autonomie multi-étapes
      • Productivité sur tâches répétitives
      Limites
      • Coût
      • Erreurs en chaîne
      • Besoin de garde-fous
      Cas d'usage
      • Refactors assistés
      • Recherche documentaire
      • Orchestration DevOps

      Conclusion

      Le Prompt Engineering est une compétence de clarification : plus votre intention est nette, plus l'IA devient utile.

      Les bonnes pratiques à garder

      • Structurer avec rôle, contexte, objectif, contraintes, format.
      • Une mission principale par prompt (ou une priorisation explicite).
      • Itérer : préciser, simplifier, ajouter des exemples, imposer le format.
      • Toujours vérifier le résultat (surtout code, sécurité, faits).

      Les erreurs à éviter

      • Demandes vagues sans public ni critères.
      • Absence de contexte projet / stack.
      • Objectifs multiples non découpés.
      • Confiance aveugle dans la première réponse.

      Méthodologie simple (5 étapes)

      1. Clarifier l'intention — Qu'est-ce qui doit être vrai à la fin ?
      2. Cadre — Rôle + contexte + contraintes non négociables.
      3. Contrat de sortie — Format, longueur, niveau de langage.
      4. Générer — Envoyer le prompt, récupérer le brouillon.
      5. Itérer & valider — Corriger, tester, documenter ce qui marche.

      Ressources pour aller plus loin

      • Documentation officielle des outils que vous utilisez (ChatGPT, Cursor, Copilot).
      • Guides d'accessibilité WCAG pour valider vos interfaces générées.
      • OWASP pour les aspects sécurité liés aux applications et aux prompts.
      • Pratique quotidienne : transformez chaque tâche réelle en prompt structuré et gardez vos meilleures versions (chapitre 11).
      Prochaine étape

      Choisissez un prompt de la bibliothèque proche de votre projet en cours, adaptez-le à votre stack, puis améliorez-le trois fois. La maîtrise vient de l'itération consciente.
